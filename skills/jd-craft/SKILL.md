# JD 质量提升与需求澄清

## 这个 skill 解决什么

JD 是招聘漏斗的**源头**——源头歪了，后面收简历、筛简历、面试全歪。

业务部门给的草稿 JD 常见三个病：
1. **泛述**（"负责相关开发工作"=什么都没说）
2. **硬性项缺失**（没写薪资范围、没写最不能妥协的技能）
3. **标题无吸引力**（平铺直叙，候选人扫一眼就划走）

本 skill 把招聘人员"和业务反复对需求、重写 JD、写问候语"的经验固化成标准流程，产出三件套。

## 三阶段工作流

```
阶段1：读取+体检     读JD → 对照6维度打分 → 标出"哪里弱"
    ↓
阶段2：产出三件套    ①需求澄清问题 ②专业版JD ③BOSS问候语
    ↓
阶段3：人工确认+落盘  用户review → 调整 → 落盘到岗位文件夹
```

## 阶段1：读取 + 体检

### 读取 JD（输入源优先级）

1. **优先** `<岗位文件夹>/jd.md`（已存24份，结构最规整，标准 markdown）
2. **fallback** `pandoc <岗位文件夹>/<岗位名>.docx -t markdown`（docx 转 markdown，保留结构）
3. **兜底** 用户直接在对话里贴 JD 文本（新岗位还没建文件夹时）

**定位岗位文件夹**：遍历 `F:/miniwanob/data/在招岗位候选人管理/**/<岗位名>/`，或用户直接给路径。
**读取命令**（docx fallback）：
```bash
pandoc "data/在招岗位候选人管理/<团队>/<岗位>/<岗位名>.docx" -t markdown
```

### JD 6维度体检

读完 JD 后，对照下表逐项评估（详见 `references/jd-quality-checklist.md`）：

| 维度 | 评什么 | 常见病 |
|------|--------|--------|
| **标题引流度** | 标题是否包含业务亮点/差异化，让候选人想点开 | 平铺直叙"XX工程师"，无业务方向 |
| **职责清晰度** | 每条职责是否具体到模块/场景，不是泛述 | "负责相关开发工作"=什么都没说 |
| **要求精准度** | 硬性要求是否可量化（年限/项目数/工具），不是"熟练" | "熟练使用Unity"→"3年以上Unity，至少1款上线项目" |
| **加分项含金量** | 加分项是否是真实差异化（不是"有责任心"这种废话） | 加分项写成软素质 |
| **结构规范度** | 是否有关于我们/职责/要求/加分项的清晰分节 | 混在一起没分节 |
| **缺失项** | 有没有漏薪资范围/工作地点/团队规模/汇报关系 | 缺薪资范围（候选人最关心的） |

**体检输出格式**：
```
=== JD 体检报告：<岗位名> ===
标题引流度：⚠️ 中（"Unity工程师"，可加"AI-Native方向"突出）
职责清晰度：✅ 良（模块清晰）
要求精准度：❌ 弱（"熟练Unity"未量化，建议改"3年+，1款上线"）
加分项含金量：⚠️ 中（有但偏软）
结构规范度：✅ 良
缺失项：❌ 无薪资范围 / ❌ 无"关于我们"（海外岗缺吸引力）
```

## 阶段2：产出三件套

### ① 需求澄清问题清单（回给业务）

**落盘**：`<岗位文件夹>/需求澄清.md`（新建，**不覆盖**评估要点.md）
**格式**：
```markdown
# <岗位名> 需求澄清

> 以下是和业务部门确认需求的关键问题，回答后用于补全优化版JD。

## 岗位定位
- 这个岗是替换某人（谁？离职？）还是新增编制？
- 最不能妥协的 3 项能力/经验是什么？（按优先级排）

## 团队现状
- 团队多大？直接汇报给谁？
- 团队现在最缺什么样的人？（补短板？扩产能？）

## 薪资与级别
- 薪资范围？（候选人最关心的，JD不写会被问爆）
- 对应职级？（初级/中级/资深/专家）

## 急迫度
- 期望什么时候到岗？
- 有备选候选人了吗？

## 特殊要求（按岗位类型，见 references/clarify-questions.md）
- [技术岗] 技术栈硬性项？必须做过什么项目？
- [设计岗] 必须的作品集方向？工具硬性要求？
- [运营岗] 必须的渠道经验？数据指标要求？
- [产品岗] 必须负责过的产品类型？海外/英文？
```

**关键**：问题要**具体到这个岗位**，不是通用模板。基于阶段1的 JD 体检结果，针对"缺失项"和"模糊点"提问。
**阻塞等用户确认**：先起草给用户看，用户说"落盘"才写文件。

### ② 专业版 JD

**落盘**：`<岗位文件夹>/jd-optimized.md`（保留原 jd.md 不动）
**格式**（标准骨架，照 UGC运营 标杆）：
```markdown
# <岗位名>（标题引流：加业务亮点/差异化方向）

## 关于我们
[团队/项目简介，2-3句，突出吸引点。缺失则让业务补]

## 岗位职责
1、[具体到模块/场景的职责，不是"负责相关开发"]
2、...
[每条职责一个清晰的动作对象，6条以内]

## 任职要求
**硬卡项**（必须满足）：
1、[可量化的硬性要求：年限/项目数/工具]
2、...

**优先项**（加分但不卡死）：
1、...
2、...

## 加分项
1、[真实差异化经验，不是软素质]
2、...
[参考JD标杆，加分项写"做过XX平台的经验""熟悉XX生态"]

## 我们提供（可选）
[薪资范围/团队亮点/成长空间]
```

**优化铁律**：
- **不编造 JD 里没有的硬性要求**——阶段2澄清问题回答后才能补全"硬卡项"的具体阈值
- 标题引流**不夸大**——"AI-Native方向""UGC生态"这类真实差异化可以加，"全球顶尖""第一"这种空话不加
- 要求分层：把原来混在一起的"任职要求"拆成**硬卡项 vs 优先项**，让候选人和 analyze-resumes 都能精确判断

### ③ BOSS 个性化问候语 + 二次追问

**落盘**：`<岗位文件夹>/boss-greeting.md`
**格式**（照 UGC运营 标杆——按候选人背景出3版 + 二次追问）：
```markdown
# <岗位名> BOSS 沟通话术

## 首次沟通问候语

### 版本1（通用版）
Hi～看你有[相关经历关键词]相关经历，我们这边正在招聘【<岗位名>】。
岗位主要围绕[1-2个核心工作内容]展开，[一句话差异化亮点]。
感觉你的经历和岗位方向挺匹配的，方便发份简历或沟通一下吗？😊

### 版本2（偏<相关经验>背景）
[针对最匹配的候选人背景定制，突出共鸣点]

### 版本3（偏<跨界但可迁移>背景）
[针对可迁移技能的候选人，说明为什么跨界也合适]

## 二次追问模板（候选人已回复但未发简历时）
您好，感谢回复～想进一步了解一下：
1、您目前是在看机会还是开放式？
2、方便发一份最新简历吗？我帮您匹配一下岗位细节～
3、您对[岗位的核心差异化，如"AI方向"/"海外业务"]怎么看？

## 使用说明
- ⚠️ **手动发送，不要批量自动发**（BOSS 容易判定违规封号）
- 问候语内容从优化版 JD 提炼，根据候选人简历背景选版本
- 二次追问在候选人回复但犹豫时用，不要连续追问压迫
```

**问候语铁律**：
- 内容**必须从优化后的 JD 提炼**（核心工作内容、差异化亮点），不是泛泛"你好，想聊聊吗"
- 让候选人**感到匹配**（点出"你的XX经历和岗位匹配"），不是群发感
- 3 个版本覆盖**不同候选人背景**（通用/最匹配/可迁移），手动选版本发

## 阶段3：人工确认 + 落盘

1. 三件套全部产出后，**先展示给用户 review**（不直接落盘）
2. 用户可逐件调整（"澄清问题加一条""JD 标题改一下""问候语版本3 太硬"）
3. 用户说"落盘"或"可以了" → 写到岗位文件夹
4. 后续：analyze-resumes 读 jd-optimized.md 做更精确的评估；interview-guide 读它出面试维度

## 不做的事（显式边界）

- ❌ **不自动发送 BOSS 问候**——封号风险，只生成文本手动发
- ❌ **不覆盖 jd.md / 评估要点.md**——原始数据只读
- ❌ **不生成简历评估**——那是 analyze-resumes 的活
- ❌ **不生成面试考核维度**——那是 interview-guide 的活
- ❌ **不编造 JD 硬性要求**——只在业务澄清回答后补全
- ❌ **不写脚本**——JD 优化是判断活，prompt 驱动，不需要机械脚本
