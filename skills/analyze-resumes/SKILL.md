# 简历评估

每天收完简历并归档后（`collect-resumes` 的下游），对当天新归档的简历逐份做 AI 筛选评估。

## 配置

所有路径走环境变量，见 `.env.example`。下文用 `$ARCHIVE_ROOT` / `$NOTES_DIR` 指代。

| 项 | 来源 |
|---|---|
| 归档根目录 | `$ARCHIVE_ROOT`（如 `F:/miniwanob/data/在招岗位候选人管理`） |
| 评估结果输出 | `$NOTES_DIR`（留空则用项目根 `notes/`） |
| 评估要点文件名 | `评估要点.md`（和 JD 同目录） |
| python | 已装 3.14 + fitz(PyMuPDF)，统一用 fitz 提取 PDF/docx |

> ⚠️ **环境提醒**：本项目只有 Explore 类型的只读 agent。写文件/改文件这类操作直接在主会话执行，不要派给 agent（只读 agent 无法写盘会 BLOCKED）。

---

## 核心定位（先读这段，建立判断基调）

这个 skill 的首要价值是**为面试做高效资源分配 + 不漏掉有潜力的人**，而不是"精确判断匹配度"。

由此推出 5 条硬约束，贯穿所有判断：

1. **判定可执行且偏宽松**：四档（强推/可推/待定/不推）必须明确。"待定"是合法常用档位，"不推"只在有明确硬伤时才用。**你怕漏人——拿不准优先落"待定"+标"建议人工复核"，不要自信地瞎判。**
2. **评估要点是活文档**：每个岗位的判断标准来自"评估要点.md"，用户可随时手动改。skill 不覆盖已确认的评估要点。
3. **分析有主次、有硬伤出局**：方向不对口一票否决（最先跑），不能被软性加分稀释成"可推"。
4. **两层产出分工**：自筛给 HR 加速（排序+判据），推荐给业务加速（亮点+风险+面试重点）。
5. **不碰薪资、不自动外发**：薪资在 BOSS 收简历时已筛过；推荐摘要只生成内容，发不发用户定。

## 阶段1：扫描归档（默认今天，可指定日期）

### 1. 定位简历目录

遍历 `$ARCHIVE_ROOT` 下所有岗位的"已收集简历/"目录，找目标日期的子目录。

> 目录命名约定：`{M.DD}_N份`（如 `6.25_1份`），和 collect-resumes 的归档一致。
> 目录名里的日期有时不规则（有的带 `_N份` 有的不带、有的只是 `M.DD`），用前缀匹配最稳。

**默认扫今天**。用户也可指定范围（如"这两天""6.24和6.25"），按用户给的日期匹配。
多个日期用多个 `-name` 模式 OR 起来：

```bash
# 单日
find "$ARCHIVE_ROOT" -type d -name "6.25*" -path "*/已收集简历/*"
# 两天
find "$ARCHIVE_ROOT" -type d \( -name "6.24*" -o -name "6.25*" \) -path "*/已收集简历/*"
```

收集到 `{岗位文件夹路径 → [简历文件路径]}`。支持 pdf/docx/doc。
**作品集文件**（文件名含"作品集""portfolio"）识别后跳过分析，只评简历本体。

### 2. 岗位定位用 find -type d，不要靠路径硬猜

岗位目录嵌套层级不统一（有的2层、有的3层），用 `find -type d -name "*岗位名*"` 定位，
不要拼路径。同名岗位可能有多个（如"Unity客户端"在长青、山海弹珠都有），按用户指定的团队筛选。

### 3. 如果没有新简历

直接告诉用户"没有新归档的简历"，结束。

---

## 阶段2：取评估要点（每个有简历的岗位都要）

对每个有简历的岗位，查其岗位文件夹根目录有没有 `评估要点.md`。

### 有 → 直接读来评

跳过起草。

### 没有 → 读 JD 起草草稿（阻塞，等用户确认）

1. 读该岗位的 JD：优先 `{岗位名}.docx`（用 fitz 提取文本，和提取简历同一方法），其次 `jd.md`
2. 起草一份 `评估要点.md` 草稿，5 个固定小标题：
   - **真实看重什么**（业务面试官最在意什么，从 JD 潜台词翻译）
   - **硬卡项**（不达标基本就不推的）
   - **可放宽项**（不够但可能可破的）
   - **雷区**（出现要标出来、影响档位的）
   - **秒推信号 / 加分项**（命中直接托到强推 / 同档往上排）
3. 输出给用户：**"这是我根据 JD 给 {岗位} 起草的评估要点，你过一遍改改？改好我就开始评。"**
4. **阻塞**：等用户确认或修改。用户确认后，将草稿写入该岗位文件夹的 `评估要点.md`。
5. 之后这个岗位每次跑都读这份，不再重新起草。

> ⚠️ 铁律：**起草是一次性的。已落盘的评估要点，skill 只读不覆盖。** 用户要调整，直接手动改文件。

---

## 阶段3：逐份提取文本 + 质检

对每份简历文件，调用提取脚本：

```bash
python scripts/extract_text.py "{简历路径}"
```

脚本输出 JSON：`{"text": ..., "char_count": N, "is_valid": bool, "issue": str|null}`

**按 is_valid 分流：**
- `is_valid=true` → 进入分析
- `is_valid=false` 且 issue 含"排版解析错乱" → **先试 recovery 模式挽救**（见下），挽救成功则进分析，仍不可读才进兜底区
- `is_valid=false`（扫描件/乱码） → 进兜底区（自筛结果的"需人工看原文件"），**不强行分析**

### recovery 模式（挽救多栏/竖排版式）

对"排版解析错乱"的简历，加 `--recovery` 参数尝试按文本块坐标重排：

```bash
python scripts/extract_text.py "{简历路径}" --recovery
```

重排后**人工/肉眼判断**是否可读（脚本的 is_valid 会是 null，需自己确认）。
可读 → 用这份文本进分析；仍碎片化 → 进兜底区。
> 不保证完美。有些重度竖排/复杂多栏连坐标重排也救不回来，这种老实进兜底区，不要拿垃圾文本硬分析。

> 多份简历可并行提取（独立无依赖），多个 Bash 调用并行发出。
> ⚠️ 中文输出写文件再 Read，不在 stdout 硬扛 GBK。脚本已用 ensure_ascii 输出 JSON，安全。

## 阶段4：逐份分析

对每份 `is_valid=true` 的简历，读 `references/analyze-prompt.md` 的提示词模板，按其 **5 维度顺序**（方向→硬卡→含金量→风险→亮点）和 **判定落档规则** 分析。

输入：岗位名 + 评估要点内容 + 简历文本。
输出：判定档位 + 一句话理由 + 判据（待定/不推必填）+ 推荐三段（仅强推/可推）。

> 完整分析指引见 `references/analyze-prompt.md`，**每次必读，不要凭记忆**。

### 多 agent 并行分析（岗位≥3 时推荐）

当待评岗位 ≥3 个时，**按岗位分派多个 Explore agent 并行分析**，平衡速度和质量：
每个 agent 拿到一个岗位的全部简历文本 + 该岗位评估要点 + analyze-prompt 模板，
独立完成该岗位所有简历的评估，返回结构化结果。

**为什么按岗位、不按简历分派**：
- 同岗位的简历要放在一起对比排序（含金量高低是相对的），拆散了失去横向参照
- 评估要点 + JD 上下文只加载一次，同岗位复用，省 token
- 岗位数通常 3-6 个，并行度正好；按简历拆会有 20+ 个 agent，开销过大且无法排序

**分派要点**：
- 每个 agent 的 prompt 必须自包含：岗位名、评估要点全文、analyze-prompt 全文、该岗位所有简历文本
- 要求 agent 返回固定结构：每份简历的 `{姓名, 判定, 一句话理由, 判据, 推荐理由, 风险提示, 建议面试重点}`
- 主会话收回各 agent 结果后，统一排序、组装两层产出
- Explore agent 是只读的——它只做"读+分析+返回结论"，不写文件，写产出由主会话统一做

> 单岗位简历 ≤3 份时不必派 agent，主会话直接分析更快。

---

## 阶段5：产出两层结果

按 `references/output-template.md` 的格式，组装结果写入 `notes/简历评估_{M.DD}.md`，同时在终端展示。

1. **第一层自筛**：按岗位分组，组内按 强推>可推>待定>不推 排序。表格先行。待定/不推附判据。
2. **第二层推荐**：只对强推/可推，每人三段（推荐理由+风险提示+建议面试重点）。
3. **兜底区**：文本异常的简历单列。

输出后告诉用户：**"评估完成，结果在 notes/简历评估_{M.DD}.md。推荐摘要可以直接复制发给业务部门。"**

> 完整格式见 `references/output-template.md`，**每次必读**。

## 阶段6：按档位分发简历（评估后必做）

评估产出落盘后，**把当天归档的简历文件按判定档位移入子文件夹**，方便用户直接打包"强推/可推"文件夹发给业务部门。

### 操作

对每个岗位当天的日期目录（如 `已收集简历/6.30_7份/`），在其中建三个档位子文件夹，把简历 `mv` 进去：

```
已收集简历/6.30_7份/
├── 强推/          ← {姓名}_{岗位简称}_{届}.pdf ...
├── 可推/
└── 待定·不推/     ← 待定和不推都放这里
```

- 每份简历移入对应档位文件夹（强推→强推/，可推→可推/，待定和不推→待定·不推/）
- 兜底区（文本异常）的简历也放"待定·不推/"，并在文件名或备注标注"需人工看原件"
- 移动用 `mv`（不是复制），避免一份简历占两处空间

### 为什么在日期目录内分、不另建"推荐给业务"区

- **日期目录是天然的"批次"边界**：6.30 收的简历评估完就在 `6.30_7份/` 内分好，发业务时整个文件夹就是这一批的结论，不会和历史批次混。
- **不污染上游**：collect-resumes 的归档逻辑和 `verify_archive.py` 数量闸门**已同步兼容档位子目录**（递归计入人头，子目录名不当条目），分档后闸门照样能跑、照样能数对份数。

### 闸门兼容说明（verify_archive.py 已改）

`verify_archive.py` 的 `collect()` 已识别档位子目录（`强推/可推/待定·不推` 等命名）：遇到这类子目录时，**递归收集子目录内的 PDF 计入人头**，子目录名本身不当作人头条目。所以分档移动后：

- 数量闸门：仍按 `_N份` 标注校验（7份标在7份的强推4+可推2+待定1=7，照常 ✅）
- 姓名/薪酬闸门：递归扫子目录内 PDF，路径会带档位前缀（如 `6.30_7份\强推\吴欣圆_...pdf`），正常校验

> 如果用户改了档位文件夹命名（如"推荐""合格"等非标准名），verify_archive 可能识别不到 → 子目录会被当散落条目。此时告知用户**档位文件夹名用标准三档名**，或同步更新脚本的 `TIER_DIR_NAMES`。

### 告诉用户

分档完成后输出：

```
已按档位分发到 {岗位}/已收集简历/{M.DD}_N份/：
  强推/ {N}人  → 可直接打包发给业务
  可推/ {N}人
  待定·不推/ {N}人
```

## 反模式（不要做）

- **不要用百分比/权重评分** — "匹配度78%"是精确的错觉，四档判定+判据才有信息量
- **不要关键词匹配当判断** — "JD有Unity简历有Unity=匹配"是幻觉，要读项目里的实际角色和深度
- **不要把简历自我标榜当真** — "主导/核心成员"可能挂名，看量化产出
- **不要维度平铺、硬伤被稀释** — 方向不对口一票否决在前，不能被软性加分冲成"可推"
- **不要把JD当金科玉律** — JD是合同价不是成交价，读评估要点的可放宽项，不够不直接判死
- **不要自信地瞎判** — 拿不准老实落"待定"+标"建议人工复核"
- **不要把待定/不推的判据藏起来** — 判据是失败反哺入口，必须附在自筛结果里
- **不要在stdout输出中文** — Windows GBK乱码，写文件再Read
- **不要每次重新起草评估要点** — 起草一次性，确认落盘后只读不改
- **不要碰薪资** — BOSS收简历时已按薪酬期望筛过
- **不要自动外发推荐摘要** — 只生成内容，发不发用户定
- **不要用 python-docx 提取 docx** — 它漏文本框内容，统一用 fitz
- **不要现场写一次性提取脚本** — 用 `scripts/extract_text.py`
- **不要拿排版错乱的文本硬分析** — 多栏/竖排简历 fitz 会按列打散成碎片，质检已拦（有效行<50%标异常），但若漏网会产出垃圾判断。可疑时先看文本是否读得通
- **不要把作品集当简历评** — 文件名含"作品集""portfolio"的跳过分析，只评简历本体
- **不要靠路径硬猜岗位目录** — 嵌套层级不统一，用 `find -type d -name "*岗位名*"` 定位
- **不要评估完就结束、不分发** — 阶段6必做：把简历按强推/可推/待定·不推移入日期目录下的档位子文件夹，用户要的是"能直接打包发业务"的结果，不是只给一份 md。分档后才能跑 verify_archive 确认没漏。
